## 2026-03-27 - [Authentication Rate Limiting]
**Vulnerability:** The authentication endpoint (`/api/auth/[...nextauth]`) lacked rate limiting, making it vulnerable to credential stuffing and brute-force attacks.
**Learning:** In NextAuth, rate limiting should ideally be handled at the middleware layer (e.g., using Redis) or via an external firewall/gateway. In-memory `Map`-based rate limiters in the `authorize` callback are a rudimentary fallback but suffer from issues in serverless environments due to stateless instance scaling and require robust cleanup to avoid memory leaks.
**Prevention:** Implement rate limiting using a durable store (like Redis or Postgres) or use Next.js Middleware in combination with a tool like `@upstash/ratelimit`. Furthermore, always normalize keys (e.g., `email.trim().toLowerCase()`) to prevent simple bypasses.
